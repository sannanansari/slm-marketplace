💾 Quantization Guide

Make Your Model 4× Smaller Without Killing Quality

Quantization reduces model weight precision from 32-bit floats to 4-bit integers. Same intelligence, fraction of the size. This guide covers GGUF, GPTQ, and AWQ — with real benchmarks and full code.

⏱ 30 min read 💻 CPU or GPU 🎯 Intermediate

What Is Quantization?

Every weight in a neural network is a number. In a standard model, each number uses 32 bits (fp32) or 16 bits (fp16/bf16). A 7B parameter model in fp16 needs 14GB just to load into memory.

Quantization reduces those numbers to 8-bit integers (INT8) or 4-bit integers (INT4). A 7B model in 4-bit needs only 3.5–4.5GB.

Precision    Bits  7B Model Size   Notes
─────────────────────────────────────────────────────────────
fp32          32    28 GB          Training precision (rare for inference)
bf16 / fp16   16    14 GB          Standard inference — good quality
INT8           8     7 GB          8-bit quantization — minimal quality loss
Q5_K_M         5     4.8 GB        GGUF 5-bit — near-lossless
Q4_K_M         4     4.1 GB        GGUF 4-bit — sweet spot ✅ recommended
Q3_K_M         3     3.1 GB        Noticeable degradation
Q2_K           2     2.5 GB        Significant quality loss
The sweet spot: Q4_K_M is the recommended starting point for almost every use case. It's 4× smaller than fp16, typically loses less than 2% on benchmarks, and runs on any modern laptop.

The Three Methods

📦
GGUF (llama.cpp)
CPU and GPU inference. Works in Ollama and LM Studio. Best for local/desktop use. Multiple quantization levels.
⚡
GPTQ
GPU-only. Post-training quantization using calibration data. Fast inference on NVIDIA. ExLlamaV2 backend.
🎯
AWQ
Activation-aware. Better quality than GPTQ at same bit-width. Best for vLLM production serving. AutoAWQ library.

GGUF — The Universal Format

GGUF (GPT-Generated Unified Format) is the file format used by llama.cpp, Ollama, LM Studio, and most local inference tools. It packages the model weights, tokenizer, and metadata into a single portable file.

Convert to GGUF with llama.cpp

bash
# 1. Clone and build llama.cpp
git clone https://github.com/ggerganov/llama.cpp
cd llama.cpp
make -j$(nproc)           # Linux/Mac
# For GPU support: make LLAMA_CUDA=1 -j$(nproc)

# 2. Install Python requirements
pip install -r requirements.txt

# 3. Download your HuggingFace model
pip install huggingface_hub
python -c "
from huggingface_hub import snapshot_download
snapshot_download('your-username/your-model', local_dir='./my-model-hf')
"

# 4. Convert to GGUF fp16 (lossless, large)
python convert_hf_to_gguf.py ./my-model-hf \
  --outfile ./my-model-f16.gguf \
  --outtype f16

# 5. Quantize to Q4_K_M (recommended)
./llama-quantize ./my-model-f16.gguf ./my-model-q4_k_m.gguf Q4_K_M

# 6. Also create Q8_0 for higher quality users
./llama-quantize ./my-model-f16.gguf ./my-model-q8_0.gguf Q8_0

# 7. Test it runs correctly
./llama-cli -m ./my-model-q4_k_m.gguf \
  -p "What is a transformer model?" \
  -n 200 --temp 0.7

# 8. Run as API server
./llama-server -m ./my-model-q4_k_m.gguf \
  --host 0.0.0.0 --port 8080 --ctx-size 4096

GGUF Quantization Levels Explained

FormatBits/weight7B SizeQuality vs fp16Use case
F161613.5 GB100% (reference)Baseline for conversion
Q8_087.2 GB~99.5%Maximum quality, GPU with 8GB+
Q6_K65.5 GB~99%High quality, 6GB GPU
Q5_K_M54.8 GB~98%Quality-focused, 6GB GPU
Q4_K_M44.1 GB~97%✅ Sweet spot — recommended default
Q4_K_S43.9 GB~96%Slightly smaller than Q4_K_M
Q3_K_M33.3 GB~93%Tight VRAM, acceptable quality
Q2_K22.6 GB~85%Extreme size constraint only
K-quants explained: The K in Q4_K_M refers to "k-quants" — a smarter quantization scheme that uses different precision for different layers. Attention layers get slightly higher precision than MLP layers. The M means "medium" (vs S=small, L=large) and refers to how many layers get the higher precision. K-quants are always better than non-K equivalents at the same file size.

Upload GGUF to HuggingFace

python
from huggingface_hub import HfApi, upload_file api = HfApi() repo_id = "your-username/your-model-GGUF" # Create the repo api.create_repo(repo_id=repo_id, repo_type="model", exist_ok=True) # Upload GGUF files for filename in ["my-model-q4_k_m.gguf", "my-model-q8_0.gguf", "my-model-f16.gguf"]: upload_file( path_or_fileobj=f"./{filename}", path_in_repo=filename, repo_id=repo_id, commit_message=f"Add {filename}" ) print(f"Uploaded {filename}") # Write a README for the GGUF repo readme = """--- license: apache-2.0 tags: - gguf - quantized - llama --- # Your Model — GGUF Versions Quantized versions of [your-username/your-model](https://huggingface.co/your-username/your-model). ## Available Files | File | Quant | Size | Quality | |------|-------|------|---------| | `my-model-q4_k_m.gguf` | Q4_K_M | 4.1 GB | ⭐⭐⭐⭐ Recommended | | `my-model-q5_k_m.gguf` | Q5_K_M | 4.8 GB | ⭐⭐⭐⭐⭐ | | `my-model-q8_0.gguf` | Q8_0 | 7.2 GB | ⭐⭐⭐⭐⭐ Max quality | ## Run with Ollama ```bash ollama run hf.co/your-username/your-model-GGUF:q4_k_m ``` ## Run with llama.cpp ```bash ./llama-cli -m my-model-q4_k_m.gguf -p "Your prompt here" -n 200 ``` """ api.upload_file( path_or_fileobj=readme.encode(), path_in_repo="README.md", repo_id=repo_id ) print("README uploaded.")

GPTQ — GPU-Optimized Quantization

GPTQ uses a calibration dataset to find optimal quantization points layer by layer. It's more accurate than basic INT4 quantization and runs natively on NVIDIA GPUs using ExLlamaV2.

bash
# Install AutoGPTQ
pip install auto-gptq optimum
python
# gptq_quantize.py from auto_gptq import AutoGPTQForCausalLM, BaseQuantizeConfig from transformers import AutoTokenizer from datasets import load_dataset MODEL_PATH = "your-username/your-model" OUTPUT_PATH = "./your-model-gptq-4bit" # 1. Load tokenizer tokenizer = AutoTokenizer.from_pretrained(MODEL_PATH) # 2. Prepare calibration dataset # Use ~128 representative samples from your domain def get_calibration_data(n_samples=128): dataset = load_dataset("wikitext", "wikitext-2-raw-v1", split="train") samples = [] for text in dataset["text"]: if len(text.strip()) > 50: tokens = tokenizer( text, return_tensors="pt", max_length=512, truncation=True ) samples.append(tokens["input_ids"]) if len(samples) >= n_samples: break return samples calibration_data = get_calibration_data() # 3. Configure quantization quant_config = BaseQuantizeConfig( bits=4, # 4-bit quantization group_size=128, # group size — 128 is standard damp_percent=0.01, # dampening factor for numerical stability desc_act=False, # activation ordering — False for faster inference ) # 4. Load model and quantize model = AutoGPTQForCausalLM.from_pretrained( MODEL_PATH, quant_config, device_map="auto" ) # This takes 5–30 mins depending on model size model.quantize(calibration_data) # 5. Save quantized model model.save_quantized(OUTPUT_PATH, use_safetensors=True) tokenizer.save_pretrained(OUTPUT_PATH) # 6. Push to HuggingFace model.push_to_hub("your-username/your-model-GPTQ", use_safetensors=True) tokenizer.push_to_hub("your-username/your-model-GPTQ") print("GPTQ quantization complete!")

AWQ — Best Quality at 4-bit

AWQ (Activation-Aware Weight Quantization) identifies which weights are most important by looking at activation magnitudes, then protects those weights with higher precision. This produces better quality than GPTQ at the same 4-bit level and is the preferred format for vLLM production serving.

bash
pip install autoawq autoawq_kernels
python
# awq_quantize.py from awq import AutoAWQForCausalLM from transformers import AutoTokenizer MODEL_PATH = "your-username/your-model" OUTPUT_PATH = "./your-model-awq-4bit" # 1. Load model and tokenizer model = AutoAWQForCausalLM.from_pretrained(MODEL_PATH, device_map="auto") tokenizer = AutoTokenizer.from_pretrained(MODEL_PATH, trust_remote_code=True) # 2. AWQ quantization config quant_config = { "zero_point": True, # Zero-point quantization "q_group_size": 128, # Group size "w_bit": 4, # 4-bit weights "version": "GEMM" # GEMM kernel (use GEMV for batch size 1) } # 3. Quantize (runs calibration automatically) # Uses MIT's PILE calibration dataset by default model.quantize(tokenizer, quant_config=quant_config) # 4. Save model.save_quantized(OUTPUT_PATH) tokenizer.save_pretrained(OUTPUT_PATH) # 5. Push to hub model.push_to_hub("your-username/your-model-AWQ") tokenizer.push_to_hub("your-username/your-model-AWQ") print("AWQ quantization complete!") # --- Test it --- from awq import AutoAWQForCausalLM model = AutoAWQForCausalLM.from_quantized(OUTPUT_PATH, fuse_layers=True) tokenizer = AutoTokenizer.from_pretrained(OUTPUT_PATH) inputs = tokenizer("What is quantization?", return_tensors="pt").to("cuda") outputs = model.generate(**inputs, max_new_tokens=100) print(tokenizer.decode(outputs[0], skip_special_tokens=True))

Which Format Should You Use?

Where will the model run?
│
├── On my laptop / desktop (no dedicated GPU)
│   └── GGUF Q4_K_M  →  run with Ollama or llama.cpp
│
├── On my laptop with GPU (RTX 3080, M1/M2 Mac)
│   ├── Want best speed?  →  GGUF Q4_K_M via Ollama
│   └── Want best quality? →  GGUF Q5_K_M or Q8_0
│
├── On a server (production API)
│   ├── Need OpenAI-compatible API at scale?
│   │   └── AWQ 4-bit  +  vLLM  (best throughput)
│   ├── Running on consumer GPU (RTX 3090/4090)?
│   │   └── GPTQ 4-bit  +  ExLlamaV2
│   └── Flexibility / compatibility?
│       └── GGUF Q4_K_M  +  llama-server
│
├── In mobile / embedded device
│   └── GGUF Q4_K_M or Q3_K_M (extreme size constraint)
│
└── Publishing for others to download
    └── Publish ALL: fp16 + GGUF Q4_K_M + GGUF Q8_0 + AWQ
        (different users have different needs)

Export GGUF Directly from Unsloth

If you fine-tuned with Unsloth, you can export directly to GGUF without installing llama.cpp separately:

python
# After fine-tuning with Unsloth (in the same Colab notebook)

# Export multiple quantization levels at once
model.save_pretrained_gguf(
    "my-model",
    tokenizer,
    quantization_method="q4_k_m"   # single level
)

# Or push multiple to Hub in one call
model.push_to_hub_gguf(
    "your-username/your-model-GGUF",
    tokenizer,
    quantization_method=["q4_k_m", "q5_k_m", "q8_0"],
    token="hf_..."
)

# Test locally with Ollama (after ollama pull)
# Ollama can load GGUF directly from HuggingFace:
# ollama run hf.co/your-username/your-model-GGUF:q4_k_m

Common Mistakes

  • Quantizing without testing — Always run your eval suite after quantizing. Some models degrade more than expected at Q4.
  • Using Q2 or Q3 for production — Below Q4, quality loss is noticeable in longer conversations. Q4_K_M is the floor for production use.
  • Forgetting to convert the tokenizer — GGUF conversion includes the tokenizer, but GPTQ/AWQ repos need the tokenizer uploaded separately.
  • Publishing only one quant level — Publish at least Q4_K_M and Q8_0. Users have different hardware. Multiple quants = more downloads.
  • Using GPTQ on CPU — GPTQ requires CUDA. For CPU inference, use GGUF.

Interview Questions

Both use 4-bit quantization. The difference is which layers get "mixed" precision. Q4_K_M (Medium) keeps more layers at slightly higher precision — specifically the attention projection layers. Q4_K_S (Small) is more aggressive. Q4_K_M is about 5% larger in file size but noticeably better in quality, especially on complex reasoning tasks.

GPTQ quantizes weights layer by layer to minimize reconstruction error without considering activation patterns. AWQ looks at which weights are activated most frequently during inference (using a calibration set) and protects those weights with higher precision. Since a small fraction of weights are responsible for most output quality, protecting the right ones produces better results.

Instead of quantizing all weights with a single scale factor, group quantization divides weights into groups (typically 128) and computes a separate scale factor per group. This preserves more information at the cost of slightly more overhead. Smaller group sizes = better quality but larger file. Group size 128 is the standard tradeoff.

Yes, and this is the standard workflow: (1) fine-tune to get the merged fp16 model, (2) quantize the merged model. Quantization is a post-training step applied to the final merged weights. Never quantize during fine-tuning — QLoRA loading in 4-bit is for memory efficiency during training, not permanent quantization.
