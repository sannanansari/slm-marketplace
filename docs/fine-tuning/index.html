Fine-Tuning Small Language Models

The complete guide to making a pre-trained SLM perform brilliantly on your specific task. Covers LoRA, QLoRA, dataset formatting, training, evaluation, and publishing — with real code for every step.

⏱ 45 min read 💻 GPU needed (free Colab) 🎯 Intermediate 📄 Full code included

What Is Fine-Tuning?

A base language model like Llama 3.2-1B has learned general language patterns from trillions of tokens of internet text. It knows grammar, facts, reasoning patterns. But it doesn't know your domain.

Fine-tuning is the process of continuing training on your specific dataset so the model becomes an expert in your domain. It's the difference between a general doctor and a cardiologist.

The core idea: Start with a smart, general model. Teach it your specific task on your specific data. Get a model that's better at your task than anything money can buy.

For 99% of use cases: QLoRA with Unsloth. It runs on a free Colab GPU and produces excellent results.

LoRA Explained — Simply

A 7B parameter model has matrices everywhere. Each attention layer has weight matrices W_q, W_k, W_v, W_o. Full fine-tuning updates all of them — billions of gradients, enormous memory.

LoRA's insight: the updates to these matrices during fine-tuning are low-rank. You don't need to update the full matrix. You can add a small adapter: two tiny matrices A and B where rank is much smaller than the full matrix.

# Full fine-tuning: update W (d × d matrix, millions of params)
W_new = W_original + ΔW

# LoRA: update via low-rank decomposition
#   A is (d × r), B is (r × d), where r << d (e.g. r=16, d=4096)
#   Only A and B are trained — that's 2 × d × r params instead of d × d

W_new = W_original + (B @ A) × (alpha / r)

# For a 4096-dim layer with r=16:
# Full: 4096 × 4096 = 16.7M params
# LoRA: 2 × 4096 × 16 = 131K params  ← 128x fewer params to train

The alpha hyperparameter controls the magnitude of the update. A common setting is alpha = 2 × r.

QLoRA — LoRA on a Quantized Model

QLoRA (Quantized LoRA) adds one more trick: load the base model in 4-bit precision (NF4 quantization). This cuts VRAM by 4× compared to fp16. Then train LoRA adapters in fp16 on top.

┌────────────────────────────────────────────────┐
│              QLoRA Architecture                 │
│                                                 │
│  Base Model (4-bit NF4 quantized, FROZEN)       │
│  ┌─────────────────────────────────────────┐   │
│  │  W_q (4-bit)  W_k (4-bit)  W_v (4-bit) │   │  ← not updated, low VRAM
│  └─────────────────────────────────────────┘   │
│         +                                       │
│  LoRA Adapters (fp16, TRAINABLE)               │
│  ┌─────────────────────────────────────────┐   │
│  │  A_q, B_q    A_k, B_k    A_v, B_v      │   │  ← updated during training
│  └─────────────────────────────────────────┘   │
│                                                 │
│  Memory: 7B model = ~4.5GB VRAM  (vs 14GB fp16) │
└────────────────────────────────────────────────┘

Step 1 — Dataset Preparation

The most common fine-tuning format is the instruction format. Each example has a system message, a user question, and an assistant answer.

Alpaca Format (simple)

json
{
  "instruction": "Classify the sentiment of the following review.",
  "input": "The product was delivered late and the quality was poor.",
  "output": "Negative"
}

ChatML Format (recommended for conversation)

json
{
  "messages": [
    {
      "role": "system",
      "content": "You are a legal contract analyst. Identify key clauses."
    },
    {
      "role": "user",
      "content": "Analyze this clause: 'Either party may terminate with 30 days notice.'"
    },
    {
      "role": "assistant",
      "content": "This is a termination clause with bilateral rights. Key details: (1) Either party can terminate — symmetric power, (2) 30-day notice period required — standard commercial practice, (3) No cause required — at-will termination. Risk: low notice period may cause operational disruption."
    }
  ]
}

Building Your Dataset Script

python
import json
import random
from datasets import Dataset

# --- Step 1: Load your raw examples ---
with open("raw_examples.json", "r") as f:
    raw = json.load(f)

# --- Step 2: Format into ChatML ---
def format_example(ex):
    return {
        "messages": [
            {"role": "system",  "content": "You are an expert in " + ex["domain"] + "."},
            {"role": "user",    "content": ex["question"]},
            {"role": "assistant", "content": ex["answer"]}
        ]
    }

formatted = [format_example(ex) for ex in raw]

# --- Step 3: Deduplicate (basic) ---
seen = set()
unique = []
for ex in formatted:
    key = ex["messages"][1]["content"][:100]
    if key not in seen:
        seen.add(key)
        unique.append(ex)

print(f"Before dedup: {len(formatted)}, After: {len(unique)}")

# --- Step 4: Train / Val / Test split ---
random.shuffle(unique)
n = len(unique)
train = unique[:int(n * 0.80)]
val   = unique[int(n * 0.80):int(n * 0.90)]
test  = unique[int(n * 0.90):]

# --- Step 5: Save as HuggingFace Dataset ---
train_ds = Dataset.from_list(train)
val_ds   = Dataset.from_list(val)
test_ds  = Dataset.from_list(test)

from datasets import DatasetDict
ds = DatasetDict({"train": train_ds, "validation": val_ds, "test": test_ds})
ds.push_to_hub("your-username/your-dataset-name")

Step 2 — The Training Script

This is a production-ready QLoRA fine-tuning script using Unsloth. Copy, modify the config block, run on Colab T4.

python
# finetune.py — QLoRA fine-tuning with Unsloth
# Install: pip install unsloth datasets trl
# Run on: Google Colab T4 (free) or any GPU with 8GB+ VRAM

# ═══════════════════════════════════
# CONFIG — edit this section only
# ═══════════════════════════════════
BASE_MODEL     = "unsloth/Llama-3.2-1B-Instruct"
DATASET_NAME   = "your-username/your-dataset-name"
OUTPUT_NAME    = "llama-1b-your-domain-lora"
HF_USERNAME    = "your-hf-username"

LORA_R         = 16      # LoRA rank. 8–64 typical. Higher = more capacity but more VRAM.
LORA_ALPHA     = 32      # Usually 2 × r
LORA_DROPOUT   = 0.05    # Small dropout prevents overfitting
TARGET_MODULES = ["q_proj", "k_proj", "v_proj", "o_proj",
                  "gate_proj", "up_proj", "down_proj"]

LEARNING_RATE  = 2e-4    # Standard for LoRA
BATCH_SIZE     = 2       # Per GPU. Increase if VRAM allows.
GRAD_ACCUM     = 4       # Effective batch size = BATCH_SIZE × GRAD_ACCUM = 8
MAX_STEPS      = 200     # For quick test. Use 500–2000 for real training.
EPOCHS         = 3       # Alternative to MAX_STEPS (comment one out)
MAX_SEQ_LEN    = 2048    # Context window. Match your data's typical length.
WARMUP_RATIO   = 0.03    # 3% of steps for warmup
# ═══════════════════════════════════

from unsloth import FastLanguageModel
from trl import SFTTrainer
from transformers import TrainingArguments
from datasets import load_dataset
import torch

# 1. Load model in 4-bit
model, tokenizer = FastLanguageModel.from_pretrained(
    model_name=BASE_MODEL,
    max_seq_length=MAX_SEQ_LEN,
    dtype=None,          # auto-detects bf16/fp16
    load_in_4bit=True,   # QLoRA: base model in 4-bit
)

# 2. Add LoRA adapters
model = FastLanguageModel.get_peft_model(
    model,
    r=LORA_R,
    target_modules=TARGET_MODULES,
    lora_alpha=LORA_ALPHA,
    lora_dropout=LORA_DROPOUT,
    bias="none",
    use_gradient_checkpointing="unsloth",  # saves VRAM
    random_state=42,
)

# 3. Load dataset
dataset = load_dataset(DATASET_NAME)
train_ds = dataset["train"]
val_ds   = dataset.get("validation", None)

# 4. Format messages into prompt strings
def format_prompt(examples):
    texts = []
    for messages in examples["messages"]:
        text = tokenizer.apply_chat_template(
            messages,
            tokenize=False,
            add_generation_prompt=False
        )
        texts.append(text)
    return {"text": texts}

train_ds = train_ds.map(format_prompt, batched=True)
if val_ds:
    val_ds = val_ds.map(format_prompt, batched=True)

# 5. Training arguments
args = TrainingArguments(
    output_dir=f"./{OUTPUT_NAME}",
    per_device_train_batch_size=BATCH_SIZE,
    gradient_accumulation_steps=GRAD_ACCUM,
    warmup_ratio=WARMUP_RATIO,
    num_train_epochs=EPOCHS,
    learning_rate=LEARNING_RATE,
    fp16=not torch.cuda.is_bf16_supported(),
    bf16=torch.cuda.is_bf16_supported(),
    logging_steps=10,
    eval_strategy="steps" if val_ds else "no",
    eval_steps=50 if val_ds else None,
    save_strategy="steps",
    save_steps=100,
    load_best_model_at_end=True if val_ds else False,
    metric_for_best_model="eval_loss",
    greater_is_better=False,
    optim="adamw_8bit",    # memory-efficient optimizer
    weight_decay=0.01,
    lr_scheduler_type="cosine",
    seed=42,
    report_to="none",      # set to "wandb" for experiment tracking
)

# 6. Trainer
trainer = SFTTrainer(
    model=model,
    tokenizer=tokenizer,
    train_dataset=train_ds,
    eval_dataset=val_ds,
    dataset_text_field="text",
    max_seq_length=MAX_SEQ_LEN,
    dataset_num_proc=2,
    args=args,
)

# 7. Train
print("Starting training...")
trainer_stats = trainer.train()
print(f"Training complete. Steps: {trainer_stats.global_step}")
print(f"Final loss: {trainer_stats.training_loss:.4f}")

# 8. Save LoRA adapter
model.save_pretrained(OUTPUT_NAME)
tokenizer.save_pretrained(OUTPUT_NAME)
print(f"Adapter saved to ./{OUTPUT_NAME}")

# 9. Push to HuggingFace Hub (adapter only — small!)
model.push_to_hub(f"{HF_USERNAME}/{OUTPUT_NAME}", token=True)
tokenizer.push_to_hub(f"{HF_USERNAME}/{OUTPUT_NAME}", token=True)
print(f"Pushed to HuggingFace: {HF_USERNAME}/{OUTPUT_NAME}")

Step 3 — Merge LoRA into Base Model

Publishing a LoRA adapter requires users to have the base model downloaded separately. For easier distribution, merge the adapter into the base model and publish the full model.

python
# merge_and_export.py — merge LoRA adapter into base model
from unsloth import FastLanguageModel

LORA_PATH   = "./llama-1b-your-domain-lora"
OUTPUT_PATH = "./llama-1b-your-domain-merged"
HF_REPO     = "your-username/llama-1b-your-domain"

# Load model with adapter
model, tokenizer = FastLanguageModel.from_pretrained(
    model_name=LORA_PATH,
    max_seq_length=2048,
    dtype=None,
    load_in_4bit=True,
)

# Merge and save as fp16 (for serving)
model.save_pretrained_merged(
    OUTPUT_PATH,
    tokenizer,
    save_method="merged_16bit",   # fp16 full model
)

# Push merged model to HuggingFace Hub
model.push_to_hub_merged(
    HF_REPO,
    tokenizer,
    save_method="merged_16bit",
    token=True
)

print(f"Full merged model available at: huggingface.co/{HF_REPO}")

# Also export GGUF for Ollama
model.save_pretrained_gguf(
    OUTPUT_PATH + "-gguf",
    tokenizer,
    quantization_method="q4_k_m"  # best balance of size/quality
)
model.push_to_hub_gguf(
    HF_REPO + "-GGUF",
    tokenizer,
    quantization_method=["q4_k_m", "q8_0"],  # publish two quants
    token=True
)
print("GGUF versions also uploaded.")

Step 4 — Test Your Fine-Tuned Model

python
# test_model.py — compare base model vs fine-tuned
from unsloth import FastLanguageModel
import torch

def load_and_test(model_path, prompts):
    model, tokenizer = FastLanguageModel.from_pretrained(
        model_name=model_path,
        max_seq_length=2048,
        dtype=None,
        load_in_4bit=True,
    )
    FastLanguageModel.for_inference(model)

    results = []
    for prompt in prompts:
        messages = [
            {"role": "system", "content": "You are a domain expert."},
            {"role": "user",   "content": prompt}
        ]
        text = tokenizer.apply_chat_template(
            messages, tokenize=False, add_generation_prompt=True
        )
        inputs = tokenizer(text, return_tensors="pt").to("cuda")

        with torch.no_grad():
            outputs = model.generate(
                **inputs,
                max_new_tokens=200,
                temperature=0.1,       # low temp for factual tasks
                do_sample=True,
                repetition_penalty=1.1
            )

        response = tokenizer.decode(outputs[0], skip_special_tokens=True)
        # Extract just the assistant's response
        response = response.split("assistant")[-1].strip()
        results.append(response)

    return results

TEST_PROMPTS = [
    "What are the key risks in a force majeure clause?",
    "How should an indemnification clause be structured?",
    "Explain the difference between warranty and representation.",
]

print("=== BASE MODEL ===")
base_results = load_and_test("unsloth/Llama-3.2-1B-Instruct", TEST_PROMPTS)
for p, r in zip(TEST_PROMPTS, base_results):
    print(f"\nQ: {p}\nA: {r[:300]}\n" + "-"*60)

print("\n=== FINE-TUNED MODEL ===")
ft_results = load_and_test("./llama-1b-your-domain-lora", TEST_PROMPTS)
for p, r in zip(TEST_PROMPTS, ft_results):
    print(f"\nQ: {p}\nA: {r[:300]}\n" + "-"*60)

Reading Your Loss Curves

The training loss should decrease consistently. The validation loss tells you whether you're overfitting.

Step  100 | Train Loss: 1.842 | Val Loss: 1.901  ← Normal (val slightly higher)
Step  200 | Train Loss: 1.654 | Val Loss: 1.698  ← Both decreasing — 
Step  300 | Train Loss: 1.512 | Val Loss: 1.557  ← Still good
Step  400 | Train Loss: 1.401 | Val Loss: 1.612  ← Val loss rising! Overfitting.
Step  500 | Train Loss: 1.289 | Val Loss: 1.743  ← Stop here, use step-300 checkpoint

Symptoms of good training:
  ✓ Train loss decreases smoothly
  ✓ Val loss decreases slightly behind train loss
  ✓ Loss curve is smooth (not spiky)

Symptoms of problems:
  ✗ Loss stays flat → learning rate too low, or data formatting wrong
  ✗ Loss spikes wildly → learning rate too high
  ✗ Val loss rises while train loss falls → overfitting (reduce epochs or add data)
  ✗ NaN loss → learning rate way too high, or NaN in data

Common Mistakes

The 7 most common fine-tuning mistakes:
  1. Wrong chat template — Each model family has a specific template (Llama-3, ChatML, Alpaca). Using the wrong one causes the model to learn the template instead of the task.
  2. Loss masking forgotten — You should only calculate loss on the assistant's tokens, not the prompt. TRL's SFTTrainer handles this automatically if you use the messages format.
  3. Too few examples — 50 examples is almost never enough. Target 500+ for simple tasks, 2000+ for complex ones.
  4. Low-quality synthetic data — If you generate training data with GPT-4 and GPT-4 makes errors, your model learns those errors.
  5. Not using eval set — Without validation loss, you can't tell if you're overfitting until it's too late.
  6. Forgetting to merge before distributing — Publishing a LoRA adapter without the base model means users need to download both and merge themselves. Publish merged too.
  7. Skipping model card — A model without a card gets zero downloads. Write the card before pushing.

Interview Questions

Full fine-tuning updates all model parameters — billions of gradients, enormous VRAM. LoRA freezes the base model and adds small low-rank adapter matrices to specific layers. Only the adapters are updated (typically 0.1–1% of total parameters). This reduces VRAM 10–100× while achieving comparable quality for most tasks.

When you fine-tune a model heavily on a narrow dataset, it can "forget" its general language ability. For example, fine-tuning on medical text might cause the model to fail at basic arithmetic. Mitigations: use LoRA (adapters don't overwrite base weights), mix general data with domain data, use fewer epochs, use a lower learning rate.

Start with r=16. If the model underfits (val loss stays high), increase to 32 or 64. If VRAM is tight, try r=8. Higher rank gives more expressive power but costs VRAM. For simple classification tasks, r=8 is often enough. For complex instruction following or reasoning, r=32–64 helps.

You want the model to learn to generate good responses, not to memorize the input prompts. Loss masking means you only compute the gradient on the assistant's response tokens, not the system/user tokens. Without loss masking, the model wastes capacity learning to reproduce prompts it will never have to generate.

QLoRA loads the base model in 4-bit NF4 quantization, which cuts VRAM ~4× vs fp16 LoRA. LoRA adapters are kept in fp16 for training precision. The key innovation is dequantization during the forward pass — the 4-bit weights are temporarily cast to fp16 for computation. Result: train a 7B model in ~6GB VRAM instead of ~24GB.
